# Contributing to SiteGuard

SiteGuard에 관심을 가져주셔서 감사합니다.

이 프로젝트는 **공개 URL 기준의 패시브 보안 점검 도구**입니다. 그래서 일반적인 웹 프로젝트보다도 기여 범위와 안전 기준을 조금 더 분명하게 두고 있습니다.

이 문서는 다음을 빠르게 이해할 수 있도록 정리한 가이드입니다.

- 어떤 종류의 기여를 환영하는지
- 어떤 변경은 이 프로젝트 방향과 맞지 않는지
- PR 전에 무엇을 확인하면 되는지
- 리뷰어가 어떤 기준으로 변경을 보는지

## 먼저 알아둘 점

SiteGuard는 다음 방향을 분명하게 유지합니다.

- 공개 URL 기준의 **패시브 외부 점검**
- 빠르고 읽기 쉬운 결과 제공
- 과장된 판정보다 **신뢰할 수 있는 해석** 우선
- 운영 가능한 수준의 안전장치 유지

즉, 이 프로젝트는 적극적인 공격 도구나 침투 테스트 프레임워크를 목표로 하지 않습니다.

다음과 같은 제안은 방향과 맞지 않을 수 있습니다.

- 인증 우회, SQL Injection, XSS를 실제로 시도하는 공격형 스캔
- 내부망, 사설 IP, `localhost` 대상으로 향하는 기능
- 서비스에 부하를 크게 줄 수 있는 공격적 크롤링 / 브루트포스
- 실제 기능보다 과장된 점수나 위험도 표현

## 환영하는 기여

다음 종류의 기여를 특히 환영합니다.

### 1. 스캐너 정확도 개선

예:

- 오탐/누락 줄이기
- 헤더, 쿠키, TLS 해석 개선
- mixed content, 리다이렉트, `security.txt` 처리 개선
- 위험도 분류 정교화

### 2. 안전성 개선

예:

- 타임아웃, 캐시, 동시성, rate limit 보강
- DNS / 네트워크 안전장치 개선
- 서버리스 환경에서의 안정성 보강

### 3. 사용자 경험 개선

예:

- 결과 화면 정보 구조 개선
- 더 자연스러운 문구 정리
- README, 문서, 예시 보강
- 접근성, 반응형, 가독성 개선

### 4. 배포 / 운영 문서 개선

예:

- Vercel / Docker / Render 문서 보완
- 환경 변수 설명 추가
- 관리 문서, 보안 문서 정리

## 기여 전에 확인할 기준

리뷰할 때는 대체로 아래 기준을 봅니다.

### 기준 1. 이 변경이 SiteGuard의 범위를 지키는가

가장 먼저 보는 기준입니다.

질문 예시:

- 이 기능이 패시브 스캐너 범위를 벗어나지 않는가?
- 허가 없는 대상에 대한 공격적 동작으로 보일 여지는 없는가?
- 공개 URL 기준의 빠른 점검이라는 정체성과 맞는가?

### 기준 2. 결과를 더 믿을 만하게 만드는가

보안 도구에서 중요한 건 “뭔가를 찾는 것”만이 아닙니다.

질문 예시:

- 오탐이 늘어나지 않는가?
- 문제를 지나치게 과장하지 않는가?
- `direct / hardening / maturity` 구분을 더 흐리지 않는가?
- 사용자에게 다음 행동이 더 분명해지는가?

### 기준 3. 운영 안전성을 해치지 않는가

질문 예시:

- 타임아웃, 바디 상한, rate limit, 동시 실행 제한과 충돌하지 않는가?
- 서버리스 / 로컬 환경에서 예기치 않은 부작용이 없는가?
- 민감한 값이나 시크릿을 코드나 문서에 노출하지 않는가?

### 기준 4. 변경 범위가 명확한가

질문 예시:

- 한 PR에 너무 많은 주제를 섞지 않았는가?
- 코드 변경과 문서 변경이 논리적으로 연결되는가?
- 관련 테스트나 설명이 같이 들어 있는가?

## 개발 환경 준비

### 요구 사항

- Node.js 20+
- npm

### 설치

```bash
npm install
```

### 개발 실행

```bash
npm run build
npm run dev
```

### 테스트

```bash
npm test
```

## 작업 방식 권장

### 작은 단위로 나누기

가능하면 하나의 PR은 하나의 주제만 다루는 편이 좋습니다.

좋은 예:

- mixed content 오탐 수정
- README 개선
- admin metrics 버그 수정
- 점수 모델 보정

조금 부담스러운 예:

- UI 전면 개편 + 스캐너 로직 수정 + 환경 변수 구조 변경

### 관련 문서도 같이 수정하기

아래에 해당하면 문서도 함께 수정해 주세요.

- 환경 변수를 추가했을 때
- API 응답 구조가 달라졌을 때
- 배포 방식이 바뀌었을 때
- 사용자가 결과를 해석하는 방식이 바뀌었을 때

대체로 아래 파일 중 하나가 같이 바뀔 가능성이 큽니다.

- `README.md`
- `README.en.md`
- `SECURITY.md`

### 테스트 가능성 고려하기

스캐너, 메트릭, 서버 동작을 바꾸는 경우엔 가능하면 테스트도 함께 추가해 주세요.

특히 아래 변경은 테스트가 있으면 좋습니다.

- 점수 / 위험도 해석 변경
- 네트워크 가드 변경
- 캐시 / rate limit 변경
- 관리자 인증 / 메트릭 변경

## 코드 / 문서 스타일

### 코드

- 가능한 한 단순하고 읽기 쉽게 작성해 주세요.
- 코드와 설정 파일은 특별한 이유가 없으면 ASCII 중심으로 유지해 주세요. 기존 파일이 이미 유니코드를 사용 중인 경우에만 필요한 범위에서 맞춰 주세요.
- 복잡한 로직은 짧은 설명 주석이 있으면 좋습니다.
- 기존과 무관한 리팩터링을 한 PR에 같이 넣지 않는 편이 좋습니다.

### 문서

- README와 사용자-facing 문구는 과장보다 명확함을 우선합니다.
- 보안 결과를 설명할 때는 단정적인 표현을 신중하게 사용합니다.
- 이 프로젝트가 **볼 수 있는 것 / 볼 수 없는 것**을 흐리지 않는 방향을 권장합니다.

## 브랜치 / 커밋

엄격한 규칙은 아니지만, 아래 정도면 충분합니다.

### 브랜치 예시

- `fix/mixed-content-parsing`
- `docs/readme-refresh`
- `feat/admin-metrics-redis`

### 커밋 메시지 예시

- `feat: refine scanner risk model`
- `fix: handle redis top domain parsing correctly`
- `docs: upgrade README and add license and security docs`

## Pull Request 체크리스트

PR 전에 아래를 한 번 확인해 주세요.

- [ ] 변경 목적이 한두 문장으로 설명된다
- [ ] SiteGuard의 패시브 범위를 벗어나지 않는다
- [ ] 관련 테스트가 있거나, 테스트가 어려운 이유를 설명했다
- [ ] `npm test`가 통과한다
- [ ] 필요한 경우 `npm run build` 결과도 확인했다
- [ ] 환경 변수 / 문서 / README 업데이트가 필요한 경우 반영했다
- [ ] 시크릿, 토큰, 개인 정보가 포함되지 않았다

UI 변경이 있다면 가능하면 전/후 스크린샷을 첨부해 주세요.

## 보안 이슈 제보

SiteGuard 자체의 취약점을 발견한 경우에는 공개 이슈 대신 [SECURITY.md](./SECURITY.md)를 먼저 확인해 주세요.

이 저장소는 제3자 사이트의 취약점을 수집하거나 공개하는 공간이 아닙니다.

## 질문이나 제안

작은 문서 수정이나 카피 수정도 충분히 좋은 기여입니다.

기여가 부담스럽게 느껴진다면, 먼저 아래처럼 작은 범위에서 시작해도 좋습니다.

- README 문장 다듬기
- 오타 수정
- 결과 문구 개선
- 환경 변수 설명 보완
- 테스트 케이스 추가

프로젝트 방향과 맞는 한, 작은 개선도 충분히 의미 있습니다.
